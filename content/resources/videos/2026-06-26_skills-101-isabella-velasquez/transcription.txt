<h3>Why skills exist</h3>

<h3>What skills are</h3>

<p><a data-t="188">In terms of what skills are,</a> <a data-t="191">at their core skills are Markdown files,</a> <a data-t="195">and Markdown is a lightweight markup language</a> <a data-t="199">with plain text formatting.</a> <a data-t="200">If you've ever used R Markdown or </a><a href="/software/quarto/">Quarto</a><a data-t="200">,</a> <a data-t="203">you have used Markdown.</a> <a data-t="206">And these files live in a folder</a> <a data-t="208">that your coding assistant can find.</a> <a data-t="211">And many, many coding assistants</a> <a data-t="214">follow the specifications for skills.</a> <a data-t="217">And so they know where to look</a> <a data-t="219">and what to do when they find these files.</a></p>

<p><a data-t="223">And even though I mentioned coding assistants specifically,</a> <a data-t="227">skills are actually supported in other sorts of programs.</a> <a data-t="230">If you use Cloud Cowork, which is a desktop app,</a> <a data-t="234">those actually support skills as well.</a> <a data-t="236">But just in the context of today</a> <a data-t="238">and this conference and us as you are users,</a> <a data-t="241">I'm going to be focusing specific on coding assistants</a> <a data-t="244">like Posit Assistant and Cloud Code.</a></p>

<p><a data-t="247">And so in terms of these files and folders,</a> <a data-t="251">they can live globally.</a> <a data-t="256">And so this means that every time</a> <a data-t="260">that you go to work with your coding assistant,</a> <a data-t="263">it'll be there installed and ready to go.</a> <a data-t="268">And skills can also be installed at a project level.</a> <a data-t="272">So if you need a specific skill for a specific project,</a> <a data-t="276">you could do that as well.</a> <a data-t="279">And it won't be available to every single project</a> <a data-t="283">that you open up.</a></p>

<h3>Anatomy of a skill</h3>

<p><a data-t="287">And so here's an example of what that looks like.</a> <a data-t="291">So if you imagine the left-hand side is my file explorer,</a> <a data-t="296">I have a project open where I have a .cloud folder,</a> <a data-t="301">within that, a skills folder,</a> <a data-t="303">within that, a shinybslib folder,</a> <a data-t="305">and then within that, a skill.nd,</a> <a data-t="308">that markdown file that defines a skill.</a> <a data-t="312">And because it's a markdown file, like a readme file,</a> <a data-t="315">you can click that and open it up</a> <a data-t="318">and take a look at what is inside.</a></p>

<p><a data-t="321">And in terms of the anatomy of a skill,</a> <a data-t="327">what actually defines a skill in this markdown file?</a> <a data-t="332">So if you have used Quarto or R Markdown before,</a> <a data-t="338">you may be familiar with this YAML header.</a> <a data-t="341">So it's denoted by three dashes.</a> <a data-t="344">And then inside those three dashes,</a> <a data-t="347">there is a key value pair.</a> <a data-t="349">So the key in this case would be name.</a> <a data-t="352">The value is shinybslib.</a></p>

<p><a data-t="356">And so the first required piece of a skill is this name,</a> <a data-t="362">which must also match the folder name</a> <a data-t="365">in which it is contained.</a> <a data-t="367">That's how the assistant knows</a> <a data-t="368">to actually go ahead and find it.</a></p>

<p><a data-t="372">And then second is the description,</a> <a data-t="375">which is probably the most important field.</a> <a data-t="378">I think this can be a little bit misleading</a> <a data-t="380">because it sounds like, oh, a description.</a> <a data-t="383">I just need to write out what exactly this,</a> <a data-t="386">what description of the skill.</a> <a data-t="388">But in fact, this skill is how the model decides</a> <a data-t="393">whether or not to load the full skill.</a> <a data-t="396">It'll read this, it'll search for keywords and context,</a> <a data-t="400">and then decide yes or no in terms of,</a> <a data-t="403">yes, I'm going to use this skill.</a> <a data-t="405">And so even though it sounds like</a> <a data-t="407">it's a description for humans,</a> <a data-t="409">it's in fact a description for the model.</a> <a data-t="412">And so it's very important to know,</a> <a data-t="417">or to write a very good description</a> <a data-t="419">so that your skill is loaded appropriately.</a></p>

<p><a data-t="424">And then below the YAML header is your actual content.</a> <a data-t="428">So these are the instructions and the worked examples,</a> <a data-t="432">the guardrails, whatever you want your model to do</a> <a data-t="436">whenever you give it a command.</a></p>

<h3>How progressive disclosure works</h3>

<p><a data-t="438">And in terms of how it works,</a> <a data-t="441">the coding assistant would load skills</a> <a data-t="443">through something that's called progressive disclosure.</a> <a data-t="447">It doesn't load every single skill up front.</a> <a data-t="450">Instead, it keeps a lightweight index</a> <a data-t="453">of just those names and descriptions.</a> <a data-t="455">And at startup, that's all that it reads.</a></p>

<p><a data-t="460">Then you give the coding agent a command,</a> <a data-t="463">like, hey, build me a </a><a href="/software/bslib/">BSLib</a><a data-t="463"> dash.</a> <a data-t="466">Hey, build me a BSLib dashboard.</a> <a data-t="469">At this point, this is when the agent would,</a> <a data-t="473">or the coding assistant would scan all those descriptions</a> <a data-t="476">and then again, look for those keywords and be like,</a> <a data-t="479">hey, does this match the command</a> <a data-t="481">that I was just given or not?</a> <a data-t="484">And again, this is meant to be just enough</a> <a data-t="486">to know whether or not the skill is relevant.</a> <a data-t="490">And only then, if it assesses, yes,</a> <a data-t="494">this skill is relevant,</a> <a data-t="496">will the coding assistant read in the full skill MD file.</a> <a data-t="500">And then at this point,</a> <a data-t="502">the coding assistant follows the different instructions</a> <a data-t="506">based on whatever is specified</a> <a data-t="509">in the actual body of the skill.</a></p>

<h3>Skills vs. agent.md</h3>

<p><a data-t="513">And you may have heard of something called agent.md</a> <a data-t="518">or claw.md.</a> <a data-t="520">And so agent is more general,</a> <a data-t="523">claw.md is specific for clawed code.</a> <a data-t="528">And there are a lot of similarities.</a> <a data-t="531">Clawed or agent.md are also markdown files</a> <a data-t="534">with instructions for your model</a> <a data-t="537">and your coding assistant in terms of telling it what to do.</a></p>

<p><a data-t="541">However, one issue of putting everything within claw.md</a> <a data-t="546">is this idea of context limits.</a> <a data-t="549">So your model can only hold so much information</a> <a data-t="553">at a particular time.</a> <a data-t="555">And if your agent.md file is really, really big,</a> <a data-t="559">it's going to hit its context limit really fast.</a> <a data-t="563">And the issue with that is as you add on</a> <a data-t="565">more and more instructions,</a> <a data-t="567">your model doesn't actually necessarily get better.</a> <a data-t="570">They can get worse.</a> <a data-t="571">They can start to drift.</a> <a data-t="574">They forget the rules that you gave it.</a> <a data-t="576">It starts to hallucinate and so on.</a> <a data-t="579">And then of course, as a user,</a> <a data-t="582">what this means is every time you are opening up a project,</a> <a data-t="587">your agent.md file is read,</a> <a data-t="589">you're using more tokens than you absolutely need,</a> <a data-t="593">which can incur further costs from you.</a></p>

<p><a data-t="596">Versus a catalog of skills,</a> <a data-t="599">which are again focused and it avoids this issue</a> <a data-t="603">by having sort of a playlist only for the task at hand.</a> <a data-t="608">They're small and composable,</a> <a data-t="610">and you can just pick them out as you need them.</a> <a data-t="614">So you and the model are really only paying</a> <a data-t="616">for what you're actually going to be using.</a></p>

<h3>Invoking skills</h3>

<p><a data-t="621">In terms of say you have your collection of skills,</a> <a data-t="624">how do you actually invoke or use your skills?</a> <a data-t="627">There's two different ways.</a> <a data-t="629">One way is explicitly.</a> <a data-t="631">So you can use what's called a slash command.</a> <a data-t="634">So a forward slash and then the name of the skill</a> <a data-t="638">to call the skill by name.</a> <a data-t="639">And you could do this if you say no,</a> <a data-t="641">that you're going to be creating a BS Lib app,</a> <a data-t="647">but more likely you will be doing it automatically.</a> <a data-t="650">So when you say, hey, build me a dashboard</a> <a data-t="653">for so-and-so thing,</a> <a data-t="655">it's going to read those names and descriptions.</a> <a data-t="658">Again, based on it's like a small index,</a> <a data-t="661">it's going to determine based on the context,</a> <a data-t="664">whether or not the skill is relevant.</a> <a data-t="666">And I'll let you know, hey, the skill is loaded.</a> <a data-t="669">You know, in this case, </a><a href="/software/shiny-r/">Shiny</a><a data-t="669"> BS Lib has been loaded.</a> <a data-t="673">And so again, it only works as well as your description,</a> <a data-t="677">which is another reason to spend a little bit of time</a> <a data-t="680">on that field.</a></p>

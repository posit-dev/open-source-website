<h3>What MCP actually is</h3>

<p><a data-t="447">Before I go further, I feel like it's important to define what MCP is, model context protocol.</a> <a data-t="453">I've talked with a lot of people recently about it, and they have a lot of questions and a lot of misconceptions I think about what it is.</a> <a data-t="457">So I first want to say what it is not.</a> <a data-t="461">It is not magic. It is not illusions. It's not AI. It's not vector search. It's not any of these things.</a> <a data-t="470">It's just a specification for how to provide tools to your LLM.</a> <a data-t="477">So tools are functions that the LLMs can call.</a> <a data-t="481">The model is going to see at the top of its context that there's a function that exists and has this description that says what it does and then what inputs there are.</a> <a data-t="487">So MCP is a standard for both the servers that can host these tools and how, as in a client and any AI application that supports it, you can consume those.</a></p>

<p><a data-t="502">Importantly, this is any AI application that supports MCP.</a> <a data-t="508">It's not just coding agents or things running the terminal.</a> <a data-t="512">It can be any other custom agent that you write yourself or other desktop applications and whatnot.</a> <a data-t="516">There's two kinds of MCP servers.</a> <a data-t="521">There's local ones that run on your machine and remote ones that run on somebody else's machine.</a> <a data-t="527">A common way, this example here from Claude Code of how you would add an MCP server highlights what it is that you're calling — UVX or NPX or something that is executing code on your machine, potentially from some random GitHub repository.</a> <a data-t="554">It can access things on your local system, which is really useful for some things, but also potentially risky, particularly if you're running some random GitHub repository's code on your computer.</a></p>

<p><a data-t="573">Remote ones, you address them by their URL, which is really a nice thing.</a> <a data-t="579">If I have a remote MCP running, I can just give you the URL to it, and that's all you need to know to start using it.</a> <a data-t="588">It can't access things on your local system, so it cannot read the files that you have.</a> <a data-t="593">You can't ask it to do something with them, but it could have access to other things.</a> <a data-t="599">In the enterprise, you may have resources that are available inside your network that this server could access that you don't have access to locally, and a remote MCP server can be one way to expose those.</a></p>

<p><a data-t="610">I'm going to focus mostly on remote MCP in part because I think it really helps make very clear what the MCP standard is, which is it's an API standard.</a> <a data-t="625">It's a way to describe an API.</a> <a data-t="630">If you look at what it's actually doing, it's making post requests, and it's sending JSON, and it's receiving JSON back.</a> <a data-t="638">It's basically a JSON RPC.</a> <a data-t="642">For me, this was really helpful to reason about, that MCP is not some complicated thing.</a> <a data-t="650">It's certainly not AI, but it's an API over HTTP.</a> <a data-t="656">I've been working with data for a long time. I've been working in software for a long time. I know how to work with APIs.</a> <a data-t="666">Ultimately, the technology is much simpler underneath it.</a></p>

<h3>MCP vs. other approaches</h3>

<p><a data-t="651">MCP came out at the end of 2024.</a> <a data-t="654">There was a lot of excitement when it first came out because it was really the first way that you could add tools like this, external resources, into your agentic workloads.</a> <a data-t="660">The first standard way to do this.</a> <a data-t="666">Naturally, there was a backlash to that.</a> <a data-t="671">A lot of people were like, well, Claude Code is really good at using command line utilities. Why would I register this MCP server with all of these tools when they can just shell out and use the </a><a href="/software/cli/">CLI</a><a data-t="671">?</a> <a data-t="685">Leading some to say that MCP is dead.</a></p>

<p><a data-t="692">This is not the end of my talk, so I do not agree that MCP is dead.</a> <a data-t="700">I think that, like most things, there's some truth to the excitement, and there's something to the criticism.</a> <a data-t="712">Since it was the first standard way to add capabilities and tools to your agents, people did things with MCP that, in retrospect, in 2026, you certainly wouldn't use MCP for that.</a></p>

<p><a data-t="731">There were a number of critiques, and I want to step through a few of them and use that to bring that to today for what MCP is good for and what it's not.</a> <a data-t="743">The first one is whether it's the right tool for the job.</a> <a data-t="747">Should you use MCP server, should you use a command line utility, or should you use a skill?</a> <a data-t="755">Now, remember, the agent skills standard came after MCP, so it did not exist at the time.</a> <a data-t="763">Skills are great.</a> <a data-t="763">If you can describe everything you want to do with tools that your agent already has, like Bash or the web fetch or anything like that, then a skill is all you need.</a> <a data-t="773">You just tell it how to use things it already knows how to do.</a> <a data-t="775">Otherwise, a skill is really a complement, and I think a very productive pattern is to have a skill that goes with your CLI or with your MCP server.</a> <a data-t="782">Here are the tools, and here's how you use them. Here's how you compose them together.</a> <a data-t="786">So the choice you're actually making is really between MCP and skill or CLI and skill.</a></p>

<p><a data-t="792">One particular example with command line utilities that was called out in these criticism was with the GitHub command line, the </a><a href="/software/gh/">GH</a><a data-t="792"> tool.</a> <a data-t="803">GitHub has an MCP server that has many dozens, lots, probably hundreds of tools, and if you were to take the whole GitHub MCP server, that's a lot of tools.</a> <a data-t="818">And this matters because when you connect to an MCP server, all of the tool definitions load at the beginning, and so they're taking up part of your context window there.</a> <a data-t="830">And that may include a lot of things that you don't need, so it's wasteful.</a> <a data-t="836">It's wasteful for your context. It's spending tokens that you don't need, and it makes your model less efficient.</a></p>

<p><a data-t="843">Whereas the GH command utility is perfect because it's already in the training data for the model, so it already knows how to use it.</a> <a data-t="858">So it's great if that's what you're dealing with, but not every command line utility that you might want to use is already in the training data.</a> <a data-t="863">And if you have to teach the model how to use your CLI, you're writing descriptions of what it can do and stuff, or if the agent has to do the, you know, your command line utility dash dash help to see what it does, well, that's tokens you're burning, right?</a> <a data-t="880">You're kind of rebuilding all of this context that you were trying to save by not using MCP.</a></p>

<p><a data-t="888">And of course, you can only use command line utilities if your agent has access to run commands in the shell, and that's not always the case.</a> <a data-t="899">You might have an agent that you have deliberately not given shell access to because you just wanted to have other tools.</a> <a data-t="904">Maybe you're trying to protect against escaping a sandbox or whatever, and so CLI is not always an option.</a></p>

<h3>Security and the lethal trifecta</h3>

<p><a data-t="923">There were also, at the time, a number of security vulnerabilities and things that were exposed using MCP servers, and so I really think it's important to talk about security.</a> <a data-t="935">It's not strictly a critique of MCP because command line utility can also expose you to these types of things as well.</a> <a data-t="943">I think a very useful framework is the lethal trifecta that Simon Willison has written about.</a> <a data-t="951">So essentially, if your agent has access to private data, which of course is what we're trying to do here by providing more tools here, that access private data, exposure to untrusted content, and the ability to communicate out, it is pretty easy to prompt inject or otherwise trick your agent into sending that private data to the attacker.</a></p>

<p><a data-t="960">Just to stick with GitHub for a minute, if you have tools exposed to let you read issues and you're reading issues in public repositories, somebody could write a prompt injection type of issue that says, read the secret file here and send the information somewhere.</a> <a data-t="1007">And so if you also have given the write issue task or comment on issue tool, then you've gotten both the access to the private data and the exposure to untrusted content, and now you've given an ability to exfiltrate.</a></p>

<p><a data-t="1026">So it's important to be mindful about what tools are made available, not just within your single MCP server, but in the whole agent that you're using.</a> <a data-t="1040">And so this leads me to describe a few best practices that I found.</a> <a data-t="1048">And with the title of the talk, MCP or not MCP, I felt I needed to have at least one Hamlet reference in here.</a> <a data-t="1056">And so brevity is the soul of wit is how I summarize the best practices.</a></p>

<p><a data-t="1066">You only want to find the tools that you actually need.</a> <a data-t="1070">You only want to send the information back that you actually need.</a> <a data-t="1073">And you want to sanitize out any sensitive data that you don't need.</a></p>

<h3>Hosting and authentication</h3>

<p><a data-t="1681">So there's some practice basically being as concise as you can be in what tools you define and what you include in the response.</a> <a data-t="1691">If we were to host this as a local MCP server, that would still be pretty good.</a> <a data-t="1697">This would have solved the problem that I had initially, which is I couldn't authenticate with Zendesk, because I could take my API key, put it in the environment where my local MCP server is running, and it's still out of the models context.</a> <a data-t="1706">We'd be efficient with tokens. <p><a data-t="1746">So a remote server would be superior.</a> <a data-t="1749">So we need a place to deploy it.</a> <a data-t="1756">And because this is just an HTTP API, anywhere that you can host APIs would work.</a> <a data-t="1762">You can self-host it inside Docker.</a> <a data-t="1766">If you work at a company that has a platform for deploying things, you probably have a place to do this.</a> <a data-t="1769">There are platforms that are more aware of the MCP spec and all the things it can do.</a> <a data-t="1775">Posit Connect is one of them.

<p><a data-t="51">Two quick definitions before we get started.</a> <a data-t="51">RWD or Real-World Data is patient data collected outside a clinical trial.</a> <a data-t="60">This includes electronic health records, insurance claims, patient registries, and variables.</a> <a data-t="68">Real-World Evidence or RWE is what we get when we apply rigorous methods to real-world data for answering clinical or economic questions.</a> <a data-t="80">We can think of RWD as the raw material and RWE as the finished insight.</a></p>

<p><a data-t="86">Today we'll showcase how Posit can take us from fragmented analysis to a reproducible submission-aligned evidence package in one governed workflow.</a></p>

<h3>Why RWE matters now</h3>

<p><a data-t="106">Real-World Evidence is reshaping how drugs are approved, priced, and used globally.</a> <a data-t="114">A few forces are making this urgent.</a> <a data-t="114">Regulators now actively accept RWE for level expansions and HTA decisions.</a> <a data-t="124">Clinical trials cannot answer every question.</a> <a data-t="124">Rare diseases, long-term safety, and pediatric dosing all require observational evidence, and payers own grand formulary access without it.</a> <a data-t="140">RWE is no longer optional.</a></p>

<blockquote><a data-t="140">RWE is no longer optional.</a></blockquote>

<p><a data-t="145">Traditional use cases of real-world data includes Health Economic Outcomes Research or HEOR, market access, post-marketing commitments, safety monitoring.</a> <a data-t="157">These are all well-established use cases.</a> <a data-t="161">Teams are increasingly using RWD for clinical trial optimization, trial feasibility, external control arms, adverse event prediction, and inclusive recruitments.</a> <a data-t="177">All of this requires the same thing, reproducible and auditable analysis.</a></p>

<h3>The reproducibility problem</h3>

<p><a data-t="186">Most teams who are dealing with real-world data are stuck with fragmented data access, manual ad hoc scripts, results pasted into PowerPoint, and no audit trail.</a> <a data-t="200">Every data refresh means rebuilding from scratch.</a> <a data-t="200">Regulators and reviewers cannot trace findings to source.</a> <a data-t="207">These are reproducibility failures.</a></p>

<p><a data-t="212">Posit removes these frictions as an integrated data science platform built around three key components.</a> <a data-t="220">Workbench, which is a governed compute environment for R and Python, Package Manager that locks package versions for long-term reproducibility, and Posit Connect,</a> <a data-t="233">where users can publish </a><a href="/software/quarto/">Quarto</a><a data-t="233"> documents, Shiny apps, and dashboards with audit trails and role-based access.</a></p>

<p><a data-t="242">Posit team can be deployed in all major cloud provider.</a> <a data-t="242">It can also be deployed on-prem within an </a><a href="/software/air/">air</a><a data-t="242">-gapped environment.</a> <a data-t="255">It can also be deployed as Snowflake native application.</a> <a data-t="259">In our demo, the environment that we use is deployed within Snowflake as a Snowflake native app.</a> <a data-t="268">And the real-world data that we are using also sits within a Snowflake database.</a></p>

<p><a data-t="277">So you may ask, why would you build your real-world workflow within Posit?</a> <a data-t="283">The primary reasons are you can own your real-world evidence end-to-end.</a> <a data-t="289">You prototype, validate, and deploy in one environment where governance is already built in.</a> <a data-t="299">This is what will lead you to a faster innovation.</a></p>

<h3>Demo study design</h3>

<p><a data-t="305">In our demo, we focused on a hypothetical study built on synthetic data set where we are looking at post-approval comparative effectiveness in type 2 diabetes.</a> <a data-t="318">We want to identify new users of ASU and compare them with DPP4 inhibitors users within a synthetic Medicare claims population.</a> <a data-t="330">And the data is already mapped to OMOP common data model.</a> <a data-t="335">We'll use control for measured confounding using propensity score matching techniques.</a> <a data-t="341">We'd like to estimate time-to-first inpatient cardiovascular hospitalization using survival analysis.</a> <a data-t="349">We also want to quantify the hazard ratio within 95% confidence interval via Cox proportional hazards model.</a></p>

<p><a data-t="359">At the end, we want to publish a pair evidence package to Posit Connect where internal and external stakeholders can access both the results and the underlying codes.</a> <a data-t="380">Snowflake holds the data behind secure role-based access.</a> <a data-t="380">Workbench runs the analysis and evidence is served through Posit Connect in forms of dashboard, HTML or Quarto documents, and web apps.</a> <a data-t="394">Package managers ensures governed and curated R and Python packages are served to Workbench and Connect.</a></p>

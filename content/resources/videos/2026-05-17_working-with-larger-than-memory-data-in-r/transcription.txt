<h3>The three layers of data analysis</h3>

<p><a data-t="460">So yeah, when we say what does that larger than memory mean, what does it entail?</a> <a data-t="460">Typically,</a> <a data-t="460">if you've worked with R before, you may have seen messages like this.</a> <a data-t="467">You cannot allocate a vector</a> <a data-t="467">of size with a very large number of gigabytes or you may even crash directly completely your R</a> <a data-t="475">Studio session, for instance.</a> <a data-t="483">And to understand what's happening, I'm trying to go back to what</a> <a data-t="483">I call the three layers.</a> <a data-t="489">The first layer is the layer with which you interact when you are doing</a> <a data-t="489">data analysis at the user interface.</a> <a data-t="496">And that's going to be what you type basically at your</a> <a data-t="496">terminal to interact with the data.</a> <a data-t="503">In the middle, there is an engine.</a> <a data-t="503">The engine is</a> <a data-t="503">really responsible for converting the instructions you gave the computer into something that it can</a> <a data-t="514">use to interact with the data that is stored either inside the hard drive or in the RAM,</a> <a data-t="525">the memory of the computer.</a></p>

<p><a data-t="532">And so to make this a little bit more concrete, imagine that you are</a> <a data-t="532">working with R and you are trying to read a CSV file.</a> <a data-t="538">Many of you said that you were working with</a> <a data-t="538">CSV files.</a> <a data-t="547">It will look like this, right?</a> <a data-t="547">You would type my underscore data is getting assigned</a> <a data-t="547">from the reader, read underscore CSV, mydata.csv.</a> <a data-t="555">The mydata.csv is a file that lives on your hard</a> <a data-t="555">drive.</a> <a data-t="561">And so when you type this, what's happening is that you are going to transfer</a> <a data-t="570">the content of this file into a representation that is going to live in the memory of your</a> <a data-t="570">computer.</a> <a data-t="580">And from there, it will be available to all to be analyzed, filtered, or whatever you</a> <a data-t="580">want to do with that data.</a></p>

<h3>Databases and the column-oriented approach</h3>

<p><a data-t="705">But so this issue is not new.</a> <a data-t="705">It's something that has existed since, maybe since the beginning of</a> <a data-t="712">computing.</a> <a data-t="712">But at least in the 70s, that's really when this problem was addressed by the creation</a> <a data-t="720">of databases.</a> <a data-t="720">If you're not familiar with this elephant, that's the logo for Postgres SQL,</a> <a data-t="727">which is a very popular database format, database system.</a> <a data-t="727">And in this case, the</a> <a data-t="737">data is going to live on your disk.</a> <a data-t="744">But when you are going to do a query, the database is</a> <a data-t="744">designed in a way where you don't need to load the entire data inside the memory of the computer to</a> <a data-t="750">get your answer.</a> <a data-t="750">The engine is able to scan the data that is on disk, decide what is helpful for</a> <a data-t="761">it to read, and it's going to transfer some of this content into the memory of the computer,</a> <a data-t="772">because that's faster to interact with rather than when it's on disk.</a> <a data-t="777">And then it will send</a> <a data-t="777">the result back to the user interface.</a></p>

<p><a data-t="785">The issue is that we are at a conference about all in</a> <a data-t="785">medicine and not SQL in medicine.</a> <a data-t="794">And so often people are familiar with all but not SQL.</a> <a data-t="794">And so</a> <a data-t="794">they may not know how to best work with this system.</a> <a data-t="803">And so that's why tools like </a><a href="/software/dbplyr/">dbplyr</a> <a data-t="803">came to be.</a> <a data-t="811">Instead of working with your data fully in memory, like in the CSV file in the</a> <a data-t="811">first example, you can use dbplyr to create a layer inside this three-layer system where you</a> <a data-t="822">are going to be able to write R code and still work with data stored inside a database.</a></p>

<p><a data-t="830">And so in this case, we can use dbplyr to create a connection to a table inside our database,</a> <a data-t="840">and we can write dbplyr syntax to try to, for instance, do a summarize by group for our data.</a> <a data-t="850">But under the hood, what's happening is that this syntax in dbplyr gets translated into SQL</a> <a data-t="858">and then gets passed on to the Postgres or the other database engine to do the data described</a> <a data-t="866">before where only some of the data that is in storage is going to be put into memory</a> <a data-t="873">to be able to materialize the answer back to the user.</a></p>

<p><a data-t="884">And so a lot of the things we're going to do today is the same idea,</a> <a data-t="891">except we are going to use DuckDB.</a> <a data-t="898">And so in this case, the syntax is going to look very similar</a> <a data-t="898">to what I just showed with Postgres.</a> <a data-t="915">We are going to use dbplyr and to some extent duckplyr to</a> <a data-t="915">write code with which we are going to interact with DuckDB as an engine that is going to look</a> <a data-t="916">at the inside of the data stored on disk and or in memory.</a> <a data-t="924">And I'll come back to that in a little bit</a> <a data-t="924">later.</a> <a data-t="935">The Arrow package works very similarly in the sense that you can still use the dbplyr</a> <a data-t="944">syntax, but under the hood, you are using the Apache Arrow engine to work with the data.</a></p>

<p><a data-t="949">So that's really a strength of the way dbplyr was designed is that from the start,</a> <a data-t="958">it was designed where you have a way to interact with data, but then you can change what is doing</a> <a data-t="958">the work.</a> <a data-t="964">So when you use dbplyr to work with DataFrame in R's memory, it's going to use some</a> <a data-t="964">code that comes with dbplyr that is going to work using R code or C++ code.</a> <a data-t="972">But there is also</a> <a data-t="972">a way to extend what dbplyr can do by connecting to other types of engine, including DuckDB,</a> <a data-t="978">Apache Arrow, or the database system that I showed earlier.</a> <a data-t="988">And this design allows us, users,</a> <a data-t="988">to always almost use the same code, even though we might be working with</a> <a data-t="993">data in different formats stored in different locations.</a></p>

<blockquote><a data-t="988">And this design allows us, users, to always almost use the same code, even though we might be working with</a> <a data-t="993">data in different formats stored in different locations.</a></blockquote>

<p><a data-t="1001">A big difference in working with DuckDB compared to that in other databases like Postgres is that</a> <a data-t="1012">internally, the data is stored on disk in a different way.</a> <a data-t="1024">The internal representation of</a> <a data-t="1024">what the data looks like inside the hard drive or the RAM of your computer is going to be very</a> <a data-t="1033">different in DuckDB compared to other database systems.</a> <a data-t="1041">DuckDB uses a column-oriented storage,</a> <a data-t="1041">which means that the data for one particular column is going to be all together.</a> <a data-t="1050">And then</a> <a data-t="1050">it will be the data for another column in your data set, and so on and so forth.</a></p>

<p><a data-t="1055">In database systems like Postgres, the data is going to be stored in a row.</a> <a data-t="1064">So the data is going</a> <a data-t="1064">to be stored as the full row of the data of the table next to each other on the hard drive.</a> <a data-t="1074">That makes a big difference because you can do things differently.</a> <a data-t="1081">It's optimized for different</a> <a data-t="1081">things.</a> <a data-t="1091">When data is stored as a column, you can do a lot of analytics operations much faster</a> <a data-t="1091">because you can only select some of the data without having to scan the entire table.</a></p>

<p><a data-t="1118">If you are trying to do a summary statistic on a particular column,</a> <a data-t="1125">when the data is stored as a row, you will need to scan the entire</a> <a data-t="1127">space of your hard drive where your data is stored to be able to access the data for each column.</a> <a data-t="1133">But when it's oriented in a column, then you can quickly scan and select all the columns at once,</a> <a data-t="1150">and so it makes it much, much faster.</a> <a data-t="1158">Additionally, because the data is not</a> <a data-t="1158">stored in the same location, it means that you can split</a> <a data-t="1164">the work and your computer can work on different columns at the same time.</a> <a data-t="1173">So you can have</a> <a data-t="1173">one part of your CPU that is going to, for instance, calculate the mean of column 1,</a> <a data-t="1179">while another part of your CPU is going to calculate the average in another column.</a> <a data-t="1186">It's not something you can really easily do when the data is stored in rows.</a></p>

<p><a data-t="1191">And so, DuckDB belongs to this category of database that is really designed for analytics.</a> <a data-t="1200">Postgres is more designed for transactional data.</a> <a data-t="1209">So you don't want to use DuckDB if you are an</a> <a data-t="1209">online shop and you try to record the orders of each people because you want to have this data</a> <a data-t="1217">organized in rows when the orders are being written.</a> <a data-t="1223">So it's fast, you write only</a> <a data-t="1223">the data one row after the other.</a> <a data-t="1231">But when you want to analyze your sales record,</a> <a data-t="1231">then you may want to switch to a different representation of your data using a column-oriented</a> <a data-t="1236">flow to be much faster in computing your summary statistics.</a></p>

<h3>Working with DuckDB: formats and storage</h3>

<p><a data-t="1263">So, to summarize what we've just covered, basically, we have the user interface layer</a> <a data-t="1263">that is really how you're going to interact with my data.</a> <a data-t="1268">You're going to have the computing engine</a> <a data-t="1268">that is going to process the data and do the computation on your data.</a> <a data-t="1277">And then you need to</a> <a data-t="1281">potentially think about where the data is going to be processed.</a> <a data-t="1282">Is it going to be</a> <a data-t="1282">only in memory, potentially only on disk, or a mix of both?</a></p>

<p><a data-t="1288">So, when we are working with DuckDB, there are many ways of working with it.</a> <a data-t="1298">And</a> <a data-t="1304">that's why it might make things a little bit confusing at first.</a> <a data-t="1304">But when you start asking,</a> <a data-t="1304">how do I interact with my data when you are working with DuckDB, you have multiple options.</a> <a data-t="1312">You can work with DuckDB writing a typical SQL statement like any other databases.</a> <a data-t="1324">But because it's a database designed for analytics, many other languages have started to</a> <a data-t="1333">use a written interface.</a> <a data-t="1333">So, you can, for instance, in R, use R code using the</a> <a data-t="1340">dplyr verb to work with the data stored inside DuckDB.</a></p>

<p><a data-t="1356">If you're using DuckDB, you are going to</a> <a data-t="1356">use a DuckDB engine to process your data.</a> <a data-t="1356">And then the last question is, where is my data?</a> <a data-t="1360">So, I wrote this way.</a> <a data-t="1360">It's called in-memory with disk spillover.</a> <a data-t="1360">What does that mean?</a> <a data-t="1361">Typically, when you're going to work with DuckDB data, you are going to either read,</a> <a data-t="1368">for instance, a Parquet file that is going to be put in memory.</a> <a data-t="1377">And then that's where the</a> <a data-t="1377">engine is going to take it to process it.</a> <a data-t="1384">But DuckDB is designed in such a way where if you're</a> <a data-t="1384">working with a very large Parquet file, a very large CSV file, you don't have to worry about</a> <a data-t="1390">running out of memory because it will work.</a> <a data-t="1398">It's like a streaming engine.</a> <a data-t="1398">It's like a movie,</a> <a data-t="1398">basically.</a> <a data-t="1398">It's only going to look at some section of the data at a time such that it won't</a> <a data-t="1405">completely fill your memory.</a> <a data-t="1405">And it's going to do just enough computation on the data it has</a> <a data-t="1412">in memory to get the answer.</a> <a data-t="1412">And DuckDB was designed in such a way that it's very efficient</a> <a data-t="1417">at doing this.</a> <a data-t="1417">And you can get answers very quickly.</a></p>

<p><a data-t="1430">So, when you're working with</a> <a data-t="1430">DuckDB, you can have data in many formats to start with.</a> <a data-t="1439">You can work with DuckDB with data</a> <a data-t="1439">stored as CSV file, Parquet, JSON.</a> <a data-t="1446">You can have even Google Sheets.</a> <a data-t="1446">There are plugins for that.</a> <a data-t="1467">And really many different formats.</a> <a data-t="1467">DuckDB also comes with its own file format that is a single</a> <a data-t="1477">file that is self-contained, that is easy to backup, easy to share with others, and that is</a> <a data-t="1484">very convenient to work with.</a> <a data-t="1492">Again, with DuckDB, you can also have your data in many different</a> <a data-t="1492">places.</a> <a data-t="1501">You can have your file on your hard drive, just like any other file, though that could be</a> <a data-t="1501">again your CSV or Parquet files or even your DuckDB files.</a> <a data-t="1507">But you can also have your CSV, Parquet,</a> <a data-t="1507">JSON files stored in the cloud in a blob storage service like AWS S3, Google Cloud, Azure.</a> <a data-t="1516">Wherever the internet can reach, you can most likely store it.</a> <a data-t="1523">DuckDB will be able to also</a> <a data-t="1523">reach these files.</a></p>

<p><a data-t="1530">And DuckDB also supports what is called a Hive partition.</a> <a data-t="1530">So, if you're working</a> <a data-t="1530">with very large datasets, sometimes it makes sense to split the datasets into multiple files</a> <a data-t="1536">following this schema that is called a Hive partition.</a> <a data-t="1541">And so, basically, you have</a> <a data-t="1552">one variable that is just like a folder system, a regular folder system.</a> <a data-t="1552">So, for instance,</a> <a data-t="1552">if you have data for multiple years, you can have one folder for each year.</a> <a data-t="1558">And within each</a> <a data-t="1558">of these folders, one file.</a> <a data-t="1564">And DuckDB will be able to read that.</a> <a data-t="1564">And so, that's very convenient</a> <a data-t="1564">because if you need to, for instance, only get data for a single year, you don't need to load</a> <a data-t="1572">the entire dataset.</a> <a data-t="1572">You can just load or scan the file that is in the year you are interested in.</a></p>

<h3>Parquet files</h3>

<p><a data-t="1586">I just want to talk a little bit more about Parquet files because</a> <a data-t="1590">that's what we're going to use today for the coding exercise.</a> <a data-t="1613">And the Parquet files are similar to what I described before with DuckDB in the sense that</a> <a data-t="1624">the layout of the data is column-oriented on disk.</a> <a data-t="1624">And that means that there's a lot of benefits.</a> <a data-t="1634">For instance, instead of having to store exactly the same data 100 times in a row,</a> <a data-t="1645">you can compress that by saying there is this data and it's going to be repeated 100 times.</a> <a data-t="1650">So, that's a way to be much more efficient in terms of storage on disk.</a></p>

<p><a data-t="1657">Another main benefit compared to, for instance, CSV file is that the data is typed.</a> <a data-t="1664">So,</a> <a data-t="1664">it means that each column has a specific data type.</a> <a data-t="1664">And so, it's much harder to introduce</a> <a data-t="1672">mistakes inside your file because if you try to insert, let's say, a numerical value</a> <a data-t="1685">in something that should be text, you could get a warning.</a> <a data-t="1695">And also, if you set your Parquet file</a> <a data-t="1695">to have specific data types, it means that everyone who is going to read this file will</a> <a data-t="1702">get the same data type, which might not be the case with CSV files depending on locals and other</a> <a data-t="1709">things.</a> <a data-t="1716">But the main benefit is that it uses the same type of layout that what, for instance,</a> <a data-t="1716">Arrow or DuckDB uses internally.</a> <a data-t="1721">And so, it's very, very fast to read this Parquet file because</a> <a data-t="1721">there is no gymnastics of trying to change the representation from an orientation in row to</a> <a data-t="1727">an orientation in column.</a> <a data-t="1727">Everything is in column and stays in column.</a> <a data-t="1732">And so, we have this very,</a> <a data-t="1732">very fast serialization from Parquet to things like Arrow or DuckDB.</a></p>
